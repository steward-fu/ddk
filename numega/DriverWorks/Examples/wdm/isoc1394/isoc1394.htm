Isoc1394 Sample

SUMMARY

The Isoc1394 sample illustrates how to set up an isochronous transfer in a 1394 driver.  The driver should be run between two PCs connected through the 1394 bus.  Each must be running a copy of the isochronous 1394 driver. 

The sample consists of a driver: isoc1394.sys, and an application: IsocApp.exe. 

The application interfaces with the driver to send read and write requests across the 1394 bus.  See Performing a read or write transaction.  It can also initiate a loopback transfer. See Setting up a loopback transfer.

 

BUILDING THE SAMPLE

The driver, isoc1394.sys, and application, isocapp.exe, may be built from the command line or from within the Visual C++ IDE.    To build from the Visual Studio IDE, open the workspace file, isoc1394.dsw.  From the build menu, select the configuration to build, and then select Build or Rebuild All to build that configuration.  Both free and checked builds are available.

To build from the command line, start a command prompt with the DDK checked or free build configuration.  Navigate to the Isoc1394 directory and type “build”.  The environment variable DRIVERWORKS must be set to the DriverWorks directory.

Note: The DriverWorks library, vdw_wdm.lib, must be built before isoc1394.sys may be built.

 

Installing the driver

The driver, isoc1394.sys, must be loaded through the device manager.  Often, when two computers are first connected, the other computer will already be listed in the Device Manager under “Other Devices” with the name “Microsoft 1394 PC.”  When the 1394 device is already listed, double click on the device, and, when the properties dialog comes up, select Update driver from the driver tab.  Choose to specify the location of the driver file, and browse to the folder containing isoc1394.inf.  The device manager should then load the driver.

If the device is not in the device manager, the driver may be loaded by using the Add New Hardware option in the control panel.

On Windows 2000 and XP, it is also possible to load the driver using the EZDriverInstaller available in the start menu under NuMega DriverStudio | Tools.

Note: To install isoc1394.sys on Windows XP, the EZDriverInstaller must be used.  Windows XP automatically loads a network driver when another PC is connected through 1394.  Because the isoc1394 driver is not in the same device class as the network driver, Windows will not allow the driver to be installed through the device manager in the normal way.

 

Note:  Isoc1394.sys needs to allocate resources from the 1394 bus on only ONE machine.  The driver cannot allocate the same resources on both machines – the first machine will be granted the resource and the second will fail to load.  For this reason, the driver defaults to not allocating resources.  Both drivers should load in this situation, but their use of resources could interfere with other 1394 devices on the bus.  To allocate resources correctly, on ONE machine, create the registry key HKLM\System\CuurentControlSet\Services\isoc1394\Parameters.  Add a DWORD value ‘AllocateResources’ set to 1.  This will tell the driver on that machine to allocate resources when it loads.

 

Performing a read or write transaction

To perform a read or write transfer, open a command prompt and type “IsocApp r 256” or “IsocApp w 256”.  Where 256 is the number of bytes to read or write. 

 

Note: The number of bytes to read or write must be a multiple of four bytes in size.  This is a limit imposed by the 1394 bus driver on isochronous transfers.

 

The second parameter “r” signifies that the application will perform a read operation.  In this case, the driver will initiate a listen operation that will not complete until the requested data becomes available on the bus. 

 

A “w” as the second parameter indicates a write operation.  Write operations complete immediately as the data is written out to the bus.  A successful completion of a write operation does not necessarily mean the data was received by a listener on the other machine.

 

To transfer data from one machine to the other, first start a read operation on one machine.  Then, start a write operation with the same number of bytes on the other machine.  The read operation will be completed with the data written by the write operation.

 

Setting up a loopback transfer

Isoc1394.sys also contains the logic necessary to perform a loopback test.   The test involves sending a packet over the 1394 bus to the other machine, and then waiting for the other machine to echo the packet back. One thousand packets are sent in each loopback test. The transfer is controlled through asynchronous commands telling the remote machine to start or stop listening for packets. 

 

The loopback transfer occurs entirely within the driver, so the application’s call will not complete until the loopback transfer completes. 

 

Start the loopback test with the command line syntax “isocapp l.”  The call will take a few seconds to complete, and then will display the loopback statistics.  The statistics display how many packets were successful, how many timed out, and how many were data errors.  A timed-out packet means that the packet was not echoed back in a reasonable amount of time.  A data error packet means that the packet echoed back was not the same as the one that was sent. 

 

The loopback transfer may be canceled before it completes.  Open a second command prompt and type “isocapp c” to cancel the loopback.  The first command prompt where the loopback was initiated should complete almost immediately after receiving the cancel command.

 

Note: The loopback transfer has a wait imposed in it between receiving a packet and sending it back.  The purpose of this wait is to make sure the driver on the other side is ready to receive the next packet.   The wait may be adjusted if the loopback is taking too long or if many packets come back with time out statuses.

 

Note:  If you plan to run a loopback transfer on both machines at the same time, you must allocate extra bandwidth in isoc1394Device::OnStartDevice.  The bandwidth allocated assumes that only one packet will be in transit at a time.

 

CODE TOUR

 

Isoc1394.sys is made up of the following classes:

Isoc1394 – This is the main driver class responsible for creating the device object.

Isoc1394Device – This is the device class.  It receives IRPs from the application and performs the correct action based on the IRP type.  The class maintains two isochronous transfer objects of type Isoc1394Transfer, one for sending data and one for receiving data.  It also has a member of type AsyncCommands, the control class for the loopback transfers.

Isoc1394Transfer – This is the transfer class, derived from K1394IsochTransfer.  It is used for both the read and write isochronous transfers.

AsyncCommands – This class is derived from K1394AsyncClient and allows the two drivers to send commands to each other in order to set up the loopback transfer
