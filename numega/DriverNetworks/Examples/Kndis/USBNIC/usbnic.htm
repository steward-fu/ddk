USBNIC NDIS Miniport Driver

[This is preliminary documentation and subject to change.]

Summary

This directory contains an NDIS Miniport Driver project files for the USBNIC card. This sample does not support any device, but rather, it is to be used as a reference driver for the development of a USB Network Interface device’s driver.

How It Works

In this sample, certain assumptions about the hardware device were made. These include:

·         A Control Pipe is used to send and receive control requests to and from the device.

·         A Send Pipe is used for the transmission of a Packet.

·         A Receive Pipe is used for the Reception of a Packet that is to be indicated to NDIS.

·         All Pipes can accept a request no larger than one Page (4Kb).

·         Firmware code must be downloaded to the device for it to operate.

·         Trigger code must be downloaded to the device so that it is activated.

·         An Interrupt Pipe is available. (For our demonstration – it is not used)

 

This sample illustrates four elements that are typically required:

·         Initialization of a USB NIC device

·         Transmission of a Packet

·         Reception of a Packet

·         Oid Control in Passive and Dispatch levels

 

Initialization of a USB NIC device

 

The Control Pipe is used to download the Firmware image to the device. In general, devices may or may not require firmware to operate. The ConfigureAndProbeDevice() member shows how to setup a device that requires a Firmware image. The function also shows how to trigger such a device to operate and also read an Ethernet configuration. Since this is a sample for a nonspecific device, it will require adjustments to the code.

 

Transmission of a Packet

 

A Packet is transmitted via the device using an IRP, an URB and a defined context. The IRP is used to carry the URB to the USB bus driver. An associated completion routine CompletionTxRoutine() is provided, that is used to free the allocated IRP and deallocate the URB. The SendComplete() member is called to tell NDIS that the packet has been successfully transmitted.

 

If the system is low on resources because there is no available IRP from the TxIrpPool, or an URB cannot be created, the Packet is queued in the m_TxQueue. When the CompetionTxRoutine() is called an attempt is made to send these Packets.

 

Reception of a Packet

 

During the Initialize() handler, a system receive area, m_pRxArea, is initialized that will be used as a holder for all receive request URBs and thier associated IRPs. The DescriptorXXXXX() members are used to create receive areas as well as reclaim/reuse receive areas.

 

Initially, DescriptorInitialize() is called once for each Rx IRP that exists in the RX IRP pool. This member creates a new URB and associates it to a newly allocated IRP. The URB is then submitted to the bus driver. When the IRP is completed, the Packet that resides in the receive descriptor (i.e. the Rfd) is completed using m_pRxArea->Complete()  which results to a call to DescriptorComplete(). The Packet is then indicated to NDIS. In Win9X environments, template class KNdisWdm9Xindication is used schedule the Packet Indication. The class uses a secure list to store the Packets to be indicated. A timer DPC is used to perform the actual indication.

 

As soon as NDIS is finished using the packet, the ReturnPacket() handler is called and the descriptor is reclaimed using m_pRxArea->Reclaim() . In the DescriptorReclaim() handler member ReSubmitRfd() is called , so that the IRP as well as the URB can be reused and resubmitted back to the bus driver for furthering the receive efforts.

 

Reference: Detailed descriptions of the KNdisWdm9Xindication and KNdisSystemReceiveArea template Classes can be found in the DriverNetworks Help.

 

Oid Control in Passive and Dispatch levels

 

An example of such an Oid handler is SetReceiveFilter(). When ConfigureAndProbeDevice() is called from Initialize(), the SetReceiveFilter() handler is called at passive level. A synchronous request is created and submitted to the bus driver. However, when NDIS calls the ::setOID_GEN_CURRENT_PACKET_FILTER() handler, member SetReceiveFilter() is called at dispatch level. Because of this, the SetReceiveFilter() member tests for the current IRQL level and functions accordingly. A completion routine is used to deallocate all the resources that were used.

Building the Driver

Open USBNIC.dsp in MSVC 6.0 IDE, select the Debug configuration, and build it. It creates files USBNIC.sys and USBNIC.nms. You will need the Windows DDK and DriverNetworks installed to successfully build the driver.

Tools

BoundsChecker for Drivers and SoftICE can be used to test some of the features of this driver.
